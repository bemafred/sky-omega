# Cycle 10 Phase 3 r4 — 21.3 B Reference + Sorted on 1.7.57 (ADR-038 + ADR-039 + MPHF instrumentation)

**Status (2026-05-13):** Cycle 10 Phase 3 r4 production validation complete. Substrate at `wiki-21b-ref-r4` loaded 21,316,531,403 triples and sealed in **23 h 56 m 50 s end-to-end** (vs cycle 9's 35 h 35 m on 1.7.50, **−11 h 38 m / −32.7 %**). First production run with the BBHash MPHF substrate (ADR-039); MPHF construction characterized at production scale via the 1.7.56 + 1.7.57 instrumentation chain. Three earlier r4-attempts aborted (1.7.53 int32 overflow, 1.7.54 BBHash non-convergence, 1.7.55 Claude-induced deploy-during-run crash) — their evidence preserved as `*-aborted-*.jsonl` for the substrate-discipline record.

## What this validation closes

- **ADR-038 P1 + P2** (merge-phase read-side: prefix-compress intermediate chunks + per-chunk frontier readahead + sidecar offset table). Production-scale merge wall-clock measured at **2 h 41 m 1 s** for 85.27 B atom occurrences over 3,923 chunks (cycle 9 didn't isolate the merge sub-phase as cleanly).
- **ADR-039** (BBHash MPHF over sealed atom set + dense final-level fallback). **First production-scale instrumented characterization** of MPHF construction on 4.005 B atoms: 25 iterative levels, dense fallback NOT engaged, placement ratio held at the BBHash theoretical 0.6065 (`1 − e^(−1/γ)` for γ=2.0), total wall-clock 54 m 29 s.
- **1.7.56 MPHF instrumentation** (`MphfBuildStartedEvent` / `MphfLevelCompletedEvent` / `MphfDenseFallbackEvent` / `MphfBuildCompletedEvent`). Plumbed through both the bulk-load finalize path (`BuildMphfFiles` ← `MergeAndWrite` ← `SortedAtomBulkBuilder.Finalize`) and the standalone `--rebuild-mphf` recovery path (via the 1.7.57 listener wire-through fix at `QuadStore.RebuildMphf`).
- **1.7.57** closes the `QuadStore.RebuildMphf` listener wire-through gap — the `--rebuild-mphf` CLI handler was passing no explicit listener and the method ignored its own instance `_observabilityListener` field. Audit confirmed this was the only category-error gap; regression test `RebuildMphf_NoExplicitListener_UsesInstanceObservabilityListener` pins the contract.

ADR-038 and ADR-039 advance from Accepted/Proposed to **Completed**.

## What this does NOT validate (open)

- **`ExternalSorter` k-way merge bypasses `BoundedFileStreamPool`.** Trigram rebuild held 8,192 chunk FileStreams open simultaneously for the full 8 h 30 m drain, peaking the process at 8,325 total FDs vs the macOS launchd effective ~10K ceiling (~17 % headroom). Didn't crash, but the unbounded fan-in is structural. Cycle 11 follow-up: extend `BoundedFileStreamPool` to back `ExternalSorter` so all rebuild paths (GPOS, trigram, any future external-sort consumer) are bounded structurally.
- **ADR-041 (cleanup-on-exception for bulk-tmp intermediates)** deferred to cycle 11. Manifested again in the 1.7.55 abort: 1.2 TB orphaned in `bulk-tmp/sorted-vocab/assigned-ids-resolver/` until manual cleanup. Third orphaning occurrence.
- **B1 sort:write attribution.** Per-spill ratio held at 6.04:1 (mean sort_ms 6,653.8 / write_ms 1,101.2) across 3,923 spills, vs the pre-B1 memory's "12–16:1" range. Candidate-not-confirmed by design — cycle 9 per-spill JSONL was not preserved, ruling out a clean head-to-head. Settled per Martin's 2026-05-11 decision; not re-litigated.

## What this validates structurally

- **MPHF construction is bounded at production scale.** ADR-039's mid-2026 design choices (γ=2.0, MaxLevels=40, MaxDenseKeys=1024) verified empirically: 25 levels with margin, dense fallback unused. The 1.7.55 dense-fallback surface is defense-in-depth that didn't need to fire at 4 B atoms; it remains the substrate's deterministic-convergence guarantee at any N.
- **The instrumentation chain works end-to-end.** Per-level events emit on the on-the-fly bulk-load path AND on the standalone `--rebuild-mphf` recovery path (the 1.7.57 fix). Future MPHF tuning (γ exploration, MaxLevels sensitivity, dense-fallback engagement at higher N) has structured telemetry to attribute against.
- **The cycle-compression methodology continues to deliver.** Cycle 10 stacked B1 + B2 + B3 + MPHF instrumentation + listener fix in one cycle (with three abort/retry attempts mid-cycle). Per-fix attribution preserved via the metric channels. Total cycle compute: ~24 h for the production validation alone; ~76 h including the three aborts. Even with the aborts, comparable to running three separate single-fix cycles.

## Substrate-discipline lessons recorded

- **`feedback_no_deploy_during_long_running_process`** (new memory entry): .NET lazy-loads referenced assemblies on first use; `dotnet tool update` mid-flight deletes the on-disk old-version directory; running process loses its not-yet-loaded `SkyOmega.Bcl`/`Mercury.Abstractions` and throws `FileNotFoundException` at the next code path that touches them. Cost cycle 10 a crash at MPHF construction after 10 h+ of parse+merge. Hard rule: NO tool updates while bulk-load/rebuild is in flight.
- **Listener wire-through audit pattern.** The 1.7.57 fix exposed a category of bug: methods that take `IObservabilityListener? listener = null` and silently ignore the instance-level listener. Audit confirmed `QuadStore.RebuildMphf` was the only such gap in the substrate; pattern documented in CHANGELOG 1.7.57.

## Information for external Wikidata-community registration

### ⚠ Dataset disclosure — full vs truthy

**This measurement is against `latest-all.ttl.bz2`, the FULL Wikidata canonical distribution. It is NOT against `latest-truthy.nt.bz2`, the truthy subset.**

This matters because **most published Wikidata benchmark numbers (QLever, Virtuoso, Blazegraph WDQS comparisons, the WDBench paper's headline numbers) run against truthy, not full.** Direct wall-clock comparison of Mercury's 23 h 57 m against a truthy-based 8 h or 12 h or 24 h published number is misleading without this disclosure.

What full contains that truthy strips:
- **Statement-level qualifiers** (e.g., point-in-time, source, reference) — a single Wikidata claim can have many qualifiers, each materializing as additional triples via `wikibase:Statement` nodes.
- **Statement-level references** (citation provenance) — every claim with a `prov:wasDerivedFrom` reference produces a `wikibase:Reference` node + its triples.
- **Sitelinks** to every Wikipedia/Wikiquote/Wiktionary/Wikisource version — `schema:about` + per-language URLs.
- The `wikibase:Statement` indirection itself (full uses `wdt:` *and* `p:`/`ps:`/`pq:` namespaces; truthy collapses to `wdt:` direct claims only).

Approximate triple counts at the time of this run (file dump 2026-04-03):
- **Full (`latest-all.ttl.bz2`)** — **21,316,531,403 triples** ← *this measurement*
- **Truthy (`latest-truthy.nt.bz2`)** — typically 12–15 B triples (we have NOT downloaded the current dump to verify; ratio of full:truthy is roughly 1.5–1.8:1 historically)

Mercury's substrate has no relationship-modeling difference between full and truthy — both are RDF triples, both go through the same parse + atom-store + B+Tree pipeline. The ratio just changes the workload size by ~1.5×. A direct truthy measurement on Mercury 1.7.57 would be a follow-up cycle if/when we want apples-to-apples vs published WDBench numbers. **This run is the substrate-capability claim** (full Wikidata loadable + queryable on one laptop in <24h), not a head-to-head benchmark claim.

The discipline rationale: Sky Omega chose to load full because (a) it's the harder workload — if the substrate handles full, it trivially handles truthy; (b) full is what Wikidata's own infrastructure (Blazegraph WDQS, now QLever) must serve in production; (c) public WDBench numbers tend to skip the heavy slice that production actually has to carry.
