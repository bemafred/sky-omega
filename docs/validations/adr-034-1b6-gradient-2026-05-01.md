# ADR-034 Phase 1B-6 Gradient — 2026-05-01

**Status (2026-05-01, Step 3 of Phase 7c Round 1):** SortedAtomStore + disk-backed AssignedIds bulk-load gradient at 1 M / 10 M / 100 M Reference scale on real Wikidata source. All three scales pass correctness (subject-bound queries on known IRIs match the Hash-backed Reference baseline) and throughput is stable at 246–297 K triples/sec.

**Critical bug fix during this run:** the SortedAtomBulkBuilder was per-batch instead of session-scoped, overwriting the vocabulary on every `CommitBatch` (the parser fires every 100 K triples). Pre-fix 1 M run produced **36 K atoms** (only the last chunk's vocab survived) with malformed query results. Fix landed in commit `3b49ce2`; post-fix 1 M produces **318 K atoms** matching the Hash baseline. Regression test added (`MultipleBatches_AccumulateAtomsAcrossSession`).

## What this gradient proves

ADR-034 Phase 1B-5d (commit `d832702`) added the disk-backed AssignedIds resolver — the implementation that lets SortedAtomStore bulk-loads scale past the in-memory `long[]` ceiling (32 GB at 1 B, 681 GB at 21.3 B). Phase 1B-6 is the gradient validation that the disk-backed path produces a correct store at 1 M / 10 M / 100 M scale on real Wikidata input.

The gradient also validates the production-path wiring (commit `7660bb4`, `3b49ce2`): `QuadStore.BeginBatch` lazy-allocates `SortedAtomBulkBuilder` with `useDiskBackedAssigned: true` once per session; `FlushToDisk` finalizes it once at session end via `FinalizeSortedAtomBulkIfPresent`.

## Memory pressure observation — input-side ceiling at 100 M

At 100 M scale, working set peaked at **39.5 GB**. The `SortedAtomBulkBuilder._atomOccurrences` is currently a `List<byte[]>` that buffers every atom occurrence in input order before `BuildExternal` consumes it. At 100 M triples × 4 atoms × ~80 bytes/atom average ≈ **32 GB heap** — matches the observed working set.

Linear extrapolation to 1 B triples: ~320 GB. Exceeds the 128 GB RAM on the M5 Max. **Step 4's 1 B trace lap will OOM** with the current design.

This was named in this session's [Phase 1B-5d retrospective](#) as `residualConcern: "input-side _atomOccurrences List<byte[]> still in-memory; 21.3B requires fixing this too before step-4 1B trace lap"`. The fix — streaming `AddTriple` directly into chunk-spill instead of buffering into a `List` — is mechanical: replace `_atomOccurrences` with a streaming `SortedAtomChunkSpiller` that calls `SortedAtomStoreExternalBuilder.SpillOneChunk` at the threshold. The architecture is correct; the scaling fix is the next ship.

For Step 3's gradient (1 M / 10 M / 100 M), 39.5 GB is well within the 128 GB host budget. The gradient passes.

## Bug discovery — multi-batch CommitBatch overwrites vocabulary

The first 1 M run produced **36,383 atoms instead of ~318 K**. Root cause: `SortedAtomBulkBuilder` was allocated per-`BeginBatch` and finalized per-`CommitBatch`. The parser fires `CommitBatch` every 100 K triples (chunk-flush boundary, `RdfEngine.LoadStreamingAsync` default), so each chunk's `Finalize` overwrote `atoms.atoms`+`atoms.offsets` with that chunk's local vocabulary. Only the last chunk's atoms survived in the final store; GSPO contained triples from all chunks but with atom IDs from each chunk's local ID-space — so queries decoded subjects as objects, etc.

The Hash-backed Reference path doesn't have this problem because `IAtomStore.Intern` is incremental — atoms accumulate naturally. ADR-034's Decision 7 ("single bulk-load contract") was being interpreted as "single CommitBatch" when it should mean "single FlushToDisk session"; the parser's chunk-flush behavior wasn't accounted for in the 1B-5b implementation.

**Fix:** session-scoped lifecycle (commit `3b49ce2`).
- `BeginBatch` lazy-allocates `_sortedAtomBulkBuilder` only on the first batch and reuses it on all subsequent batches
- `CommitBatch` is a near no-op for Sorted-backed Reference: it clears the active-batch flag and lets the bulk-builder accumulate
- `FlushToDisk` calls `FinalizeSortedAtomBulkIfPresent` once at session end — finalizes vocabulary, replays buffered triples into GSPO sorter

Restores symmetry with the Hash path's `_bulkSorter` lifecycle (which already persisted across batches and was drained once at `FlushToDisk`).

Regression test: `SortedAtomStoreBulkLoadTests.MultipleBatches_AccumulateAtomsAcrossSession` — three distinct batches each contributing 3 atoms, expects 9 atoms in the final store. Pre-fix code would have produced 3.

## What this validates / does not validate

Validates:
- ADR-034 Phase 1B-5d disk-backed AssignedIds path produces correct atom IDs at 1 M / 10 M / 100 M Reference scale
- Production-path wiring (`useDiskBackedAssigned: true` always-on in `QuadStore.BeginBatch`)
- Session-scoped `SortedAtomBulkBuilder` lifecycle survives multi-batch chunk-flush from the parser
- Throughput holds steady at 246–297 K triples/sec across two orders of magnitude

Does NOT validate:
- 1 B+ scale — the input-side `_atomOccurrences` ceiling (~32 GB heap at 100 M, ~320 GB extrapolated at 1 B) requires the streaming-input fix before Step 4's 1 B trace lap can run on a 128 GB host
- Query equivalence at the FULL triple level — spot-check on known IRIs verified, but a complete byte-for-byte comparison vs Hash output is not performed (atom-ID assignment differs by design between the two paths; ADR-034 Decision 5)
- Rebuild path — secondary indexes (GPOS, trigram) not built; gradient is bulk-load only. Phase 7a metrics infrastructure exercises the rebuild side.

## Consequences

1. **ADR-034 Phase 1B-6 gradient passes at 1 M / 10 M / 100 M.** Disk-backed AssignedIds is correct on the bulk-load path.
2. **The lifecycle bug surfaced as a correctness issue, not just a performance one.** Pre-fix the 1 M run produced a working store with WRONG triples; query results decoded literals as subjects. The fix and regression test prevent recurrence.
3. **Step 4 (1 B trace lap) is blocked by the input-side ceiling.** The streaming-input fix is mechanical (replace `List<byte[]>` with chunk-spilling spiller; pattern already exists in `SortedAtomStoreExternalBuilder.SpillChunks`). Estimated half-day of work.
4. **Step 3 deliverable closed.** The gradient artifacts are at `/tmp/round1-gradient/{1m,10m,100m}.jsonl` and the stores at `/Users/bemafred/Library/SkyOmega/stores/wiki-{1m,10m,100m}-r1/`.
