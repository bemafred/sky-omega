# ADR-034 Step 4 — 1 B Reference + Sorted Trace Lap — 2026-05-01

**Status:** Phase 7c Round 1 Step 4 complete. SortedAtomStore + disk-backed AssignedIds + streaming-input bulk-load validated end-to-end at 1 B Reference scale on real Wikidata. Total wall-clock 1 h 0 m 36 s. Validates the int64 widening from commit `bd2f9c1`.

**Critical bug fix during this step:** the first 1 B run (commits through `ba90595`) loaded the 1 B triples and finalized the vocabulary correctly, but produced an empty GSPO index. Root cause: `int globalIdx` in `SortedAtomBulkBuilder` and `SortedAtomStoreExternalBuilder` overflowed past int32 max (~2.1 B) at 1 B triples × 4 atoms = 4 B occurrences. Wrapped negative InputIdx values broke the dense-coverage invariant in `DiskBackedAssignedIds.DiskBackedReader`, throwing mid-way through `EnumerateResolved`. The exception killed `FlushToDisk` before `_bulkSorter` drained into GSPO. Fix: int32 → int64 widening throughout the input-index path (commit `bd2f9c1`); chunk file format header bumped from 8 bytes to 12 bytes per record. Regression test (`DiskBacked_ChunkFile_RoundTripsInt64InputIdx`) added.

## What this validates

- **Phase 1B-5d disk-backed AssignedIds at 1 B scale.** 4 B occurrence resolution records flow through `ExternalSorter<ResolveRecord>`. Memory bounded by chunk-buffer threshold throughout.
- **Phase 1B-5e streaming input.** The previous in-memory `List<byte[]>` ceiling (~32 GB heap at 1 B) is gone. GC heap stayed at ~1 GB throughout; working set ~1.2 GB.

## Bug arc — int32 overflow

The first 1 B run (commits up to `ba90595`) produced:

- 213 M atoms in atoms.atoms ✓
- atoms.offsets 1.7 GB ✓
- **gspo.tdb 64 blocks (empty)** ✗
- bulk-tmp 217 GB of leaked occurrence chunks ✗
- No "Load complete" output, no `load.summary` metric — process exited silently after vocab finalize

Diagnosis path:
1. Process exit code 0 from the polling wrapper (not Mercury); investigated stdout, found no completion message.
2. Examined store files — atoms intact, GSPO empty. Confirmed FlushToDisk ran far enough to write atoms.atoms but not far enough to populate GSPO.
3. Calculated: 1 B triples × 4 atoms = 4 B occurrences. int32 max = 2.147 B. Globally-incremented `int globalIdx` wraps to negative around the 537 M-triple mark.
4. `DiskBackedAssignedIds.DiskBackedReader` asserts dense coverage: `if (rec.InputIdx != _expectedIdx) throw`. The wrap produces records with negative InputIdx that violate the assertion. Throws inside the `EnumerateResolved` iterator, killing `FlushToDisk`.

Fix (commit `bd2f9c1`): widen `int → long` throughout the input-index path:
- `SortedAtomBulkBuilder._globalIdx`
- `SortedAtomBulkBuilder._spillBuffer` tuple type
- `SortedAtomStoreExternalBuilder.SpillChunks` return + locals
- `SortedAtomStoreExternalBuilder.SpillOneChunk` buffer type
- `SortedAtomStoreExternalBuilder.MergeAndWrite` parameter
- `SortedAtomStoreExternalBuilder.ChunkReader.Current` tuple type
- Chunk file format header: int32 length + int32 InputIdx (8 bytes) → int32 length + int64 InputIdx (12 bytes)

Regression test (`DiskBacked_ChunkFile_RoundTripsInt64InputIdx`) verifies the file format round-trips InputIdx values past int32 max. Runs in 9 ms; would have caught the bug at design time.

## Lesson — scale-boundary tests are not optional

Three iterations now of the same shape — small-scale validation passes, large-scale exposes architectural boundaries:

- **Phase 5.2 (2026-04-21):** dotnet-trace + iostat at 100 M revealed wall-clock equality was hiding 453 s GC.RunFinalizers + 552 s Monitor.Enter_Slowpath that didn't exist in the sequential baseline.
- **Phase 1B-5b multi-batch lifecycle (this session, 2026-04-30):** unit tests at single-batch passed. Production at 1 M Wikidata triggers parser chunk-flush every 100 K triples — multi-batch path unexercised, vocabulary overwrites silently.
- **Phase 1B-5e int32 overflow (this step, 2026-05-01):** unit suite at 20 K triples (80 K occurrences) and gradient at 100 M (400 M occurrences) sit inside int32. Only 1 B production (4 B occurrences) crosses the boundary.

Generalizable rule: when designing a counter or invariant for a substrate that targets a known-large scale, audit the headline-scale value at design time. For Sky Omega's Reference profile targeting 21.3 B Wikidata (85 B atom occurrences), data-dependent counters MUST be int64. Add focused regression tests that exercise the boundary explicitly — they don't require running the full workload, just hand-crafted records that exercise the limit.

## What this validates / does not validate

Validates:
- Phase 7c Round 1 Steps 1, 1B-5d, 1B-5e, 3, and 4 all green at 1 B Wikidata Reference
- Disk-backed AssignedIds correctness at 4 B occurrence scale
- Streaming input scales linearly (memory bounded by chunk-buffer regardless of input)
- Session-scoped lifecycle survives multi-batch chunk-flush
- int64 InputIdx round-trips correctly through chunk files

Does NOT validate:
- 21.3 B scale — linear extrapolation from this 1 B run suggests parser ~13 h + FlushToDisk ~8.5 h ≈ 21.5 h bulk-load. Plus rebuild (GPOS + trigram). Phase 6's Hash baseline was 85 h end-to-end including 11.65 h rebuild; if the linear scaling holds, Sorted at 21.3 B is ~30-35 h end-to-end (not counting rebuild) — a meaningful improvement, but only validated by running it.
- Rebuild path on Sorted-backed Reference — gradient was bulk-load only.
- Query-side performance against the Sorted store — separate WDBench-class measurement.

## Consequences

1. **Phase 7c Round 1 Step 4 closes.** The 1 B trace lap completed successfully; the GSPO is queryable; correctness checks all green.
2. **Step 5 (21.3 B headline run) is unblocked.** All known architectural ceilings on the Sorted bulk-load path are removed: input-side (streaming), output-side (disk-backed AssignedIds), arithmetic (int64 InputIdx), lifecycle (session-scoped).
3. **The int64 widening fix has no production hazard at smaller scales.** Chunk files now use 12-byte headers (was 8); slight increase in transient disk during merge (~50 % more header bytes per record). At 1 B scale this is negligible against the multi-GB chunk volumes.
4. **Phase 6 Hash baseline at 85 h gets a real comparator.** Sorted's projected 30-35 h at 21.3 B (excluding rebuild) is the architectural alternative ADR-034 targeted; the value of alphabetically-dense IDs is downstream prefix compression + bit-packed atom IDs in Round 2.
5. **Three "scale-boundary tests not optional" patterns recorded** in Mercury (`obs:scale-boundary-tests-not-optional`). For Round 2 design (prefix compression + bit-packed IDs), pre-emptively audit any data-dependent counters against 85 B occurrences (full Wikidata) and 4 B atoms.
