# ADR-030 Phase 5.2 dotnet-trace Validation — 2026-04-21

**Status:** Phase 5.2 profile pass on Reference 100M rebuild via `dotnet-trace --profile dotnet-sampled-thread-time`. Result: **the rebuild is not compute-bound on Mercury managed code.** Total managed-code inclusive time across the hot path is ~150 s out of a 559.6 s wall-clock window. The remaining ~400 s is in `UNMANAGED_CODE_TIME` — kernel transitions, mmap fault servicing, GC suspension, lock-acquire slow paths. None of the four candidate hot spots from the Phase 3 follow-up list are large enough to matter.

## Setup

- Mercury 1.7.37 (commit 92afd23)
- Store: `~/Library/SkyOmega/stores/wiki-100m-ref` (Ready state, 100M Reference profile)
- Command: `mercury --store wiki-100m-ref --rebuild-indexes --no-http --no-repl`

## What the trace cannot tell us

`dotnet-sampled-thread-time` samples the managed call stack of every thread, regardless of whether the thread is actually running on a CPU. When a thread blocks in unmanaged code — mmap fault, native semaphore wait, GC suspension — the sample shows the topmost managed frame plus `UNMANAGED_CODE_TIME`. Three causes are indistinguishable in this view:

- **Lock contention** in user code (e.g. consumer waking up to dequeue from BroadcastChannel)
- **GC suspension** (the runtime stop-the-world to allow finalizers/collection)
- **Kernel I/O** (mmap page-fault servicing on first touch of a freshly-mapped index region)

The 508.5 s of `Monitor.Enter_Slowpath` attributed to `ConsumeReferenceIntoTrigram` is the strongest concrete signal but ambiguous: it could be (a) the consumer genuinely contending on a lock 95% of its wall time, or (b) the consumer parked in `Monitor.Wait` for the next batch and the `Slowpath` frame appearing because the wake-and-reacquire goes through it. The trace mode cannot distinguish these.

Similarly, the 362 s of `PollGCWorker` says the runtime is suspending workers for safepoints, but does not say how long suspensions actually take.

## Implications

**1. The four candidate hot spots from the Phase 3 follow-up list are real but small.**

The pickup memory called out: BroadcastChannel lock acquisitions (~200M), trigram `GetAtomSpan`/`IndexAtom` (~46M), GSPO B+Tree enumerator `MoveNext` (~99M), GPOS sort-insert. The trace shows:

- Trigram indexing combined: ~75 s (13% of wall)
- GPOS sort: 80.6 s (14% of wall)
- B+Tree page operations: ~4 s (negligible)
- BroadcastChannel: cannot be measured as a discrete cost in this profile

Even if all four were eliminated, the rebuild would only drop from 559 s to ~400 s — and 400 s of "kernel time we can't see" remains.

**2. The "external sort + parallel sort + radix keys" follow-up list won't unblock 21.3 B alone.**

The sort itself is 80 s out of 559 s. Even a 5× sort speedup recovers ~64 s, ~12% of the run. That's not the path to a 24 h Phase 6 exit at 21.3 B.

**3. GC pressure is the underexplored cost.**

453 s on the finalizer thread plus 362 s of `PollGCWorker` across worker threads says the runtime is doing significant collection/finalization work. This is unaccounted for in any prior validation. Likely sources:

- Per-broadcast batch object allocations (every fanout from producer creates managed objects on both consumers' channels)
- String allocations in the trigram path (`ChangeCaseCommon` allocates a new String per literal — 5.9 s exclusive at 100M scales linearly)
- Posting list resize allocations (`EnsurePostingCapacity`)

**4. The architectural cost is in the cross-thread channel pattern itself.**

If the Trigram consumer truly spends 95% of wall in lock-acquire-wait cycles (which is one valid reading of the 508 s), then the BroadcastChannel design — producer fans 99M entries to 2 consumers via Wait/Notify — is paying coordination overhead per entry rather than per batch. Larger batches would amortize, but the parallel rebuild Phase 5.1.b validation already showed parallelism is neutral at this scale.

## What this trace mode cannot resolve

To split the ~400 s of `UNMANAGED_CODE_TIME` into its real sub-causes, we need a profile mode that includes:

- **GC pause timing** — `dotnet-trace --profile gc-verbose` or `dotnet-common` with GC events would let us subtract suspension time from the per-thread budget
- **Native call attribution** — Instruments on macOS would show kernel time spent in mmap fault servicing vs futex waits vs page table operations
- **Lock contention** — `--clrevents contention` would emit explicit contention events with held/wait timing

A second trace pass with one or more of these is the obvious next move.

## A/B comparison vs 1.7.34 sequential baseline

Same trace command run against a fresh 1.7.34 build (commit be91cb2) on the same store. 1.7.34 is the post-Decision-5, pre-parallel sequential rebuild path. Phase 5.1.c reported wall-clock as "neutral" across {sequential 512 s, parallel-random 524 s, sort-insert 511 s}. That call was wall-clock-only.

| Metric | 1.7.34 (sequential) | 1.7.37 (parallel + sort) |
|---|---:|---:|
| Wall-clock (s) | 512.9 | 559.6 |
| Threads sampled | 1 | 15 |
| Sum active thread-time (s) | 512.9 | **5314.7** |
| `UNMANAGED_CODE_TIME` (s) | 512.9 | **5311.2** |
| `GC.RunFinalizers` (s) | **0** | 453.5 |
| `Thread.PollGCWorker` (s) | **0** | 362.3 |
| `Monitor.Enter_Slowpath` (s) | **0** | 552.2 |
| `Monitor.Wait` (s) | **0** | 559.6 |
| `LowLevelLifoSemaphore.Wait` (s) | **0** | 2504.9 |
| `ManualResetEventSlim.Wait` (s) | **0** | 794.5 |
| `RebuildReferenceSecondaryIndexes` inclusive (s) | 512.6 | 559.1 |

Six categories of cost in 1.7.37 are **literally absent** in 1.7.34 — zero samples ever land in those frames. 1.7.34 generates so little garbage that no finalizer thread ever wakes up; spawns no threadpool workers; takes no Monitor slow paths.

### What actually happened in Phase 5.1.b/c

**Got:** GPOS sort-insert (replaces random B+Tree walks), parallel trigram building (overlaps with GPOS).

**Paid:**
- BroadcastChannel + per-batch allocations on the producer
- 14 extra threads (stacks, TLS, scheduler involvement)
- Allocation pressure: 453 s of finalizer-thread work
- 362 s of `PollGCWorker` — runtime constantly suspending workers for safepoints
- 552 s of `Monitor.Enter_Slowpath` on the trigram consumer (parked ~95% of wall, lock-reacquire cost on every wake)

The compute savings and the new overhead net to zero on wall-clock at 100M because the M-series CPU has spare cores and the M5 Max has 128 GB RAM to absorb the allocation pressure.

### Caveat on Mercury managed-frame inclusive comparisons

In 1.7.34 the trigram inclusive numbers are large (`IndexAtom` 448.5 s, `AddToPostingList` 256 s, `ChangeCaseCommon` 187.1 s) because the single thread is genuinely executing those frames most of its wall time. In 1.7.37 the trigram consumer is parked most of the time, so samples catch it on `Monitor.Enter_Slowpath` (29.5 s `IndexAtom` inclusive). **Same trigram work is happening; we just sample it less often because the thread is mostly idle.** The Mercury frame inclusive deltas are sampling artifacts, not workload changes.

### Why this matters for 21.3 B

- GC pressure scales linearly with N. 453 s at 100M extrapolates to ~96 000 s (~27 hours) at 21.3 B if linear.
- Lock chatter scales linearly with batches.
- mmap fault thrashing scales worse than linearly past RAM. More concurrent threads = more concurrent random page accesses = worse page-cache hit rate. 21.3 B Reference will be ~2.3 TB on disk, far beyond RAM.
- The "spare cores absorb the overhead" effect breaks down when memory bandwidth or page cache becomes the bottleneck.

The single-thread 1.7.34 path has none of these scaling exposures.

## I/O measurement — 10M Reference rebuild on 1.7.34

To distinguish three candidate causes for `UNMANAGED_CODE_TIME` (real disk I/O, in-cache page-fault overhead, lock waits), the 1.7.34 binary was run against `wiki-10m-ref` while `iostat -d -w 1 disk0` and `vm_stat 1` captured per-second disk and paging activity.

| | Cold rebuild | Warm rebuild (back-to-back) |
|---|---:|---:|
| Wall-clock | 28.18 s | 28.51 s |
| Peak iostat MB/s | 327 | 328 |
| Peak iostat tps | 12,059 | 11,926 |
| Peak vm_stat pageins | 21,024 pages/s = 336 MB/s | 21,154 pages/s = 338 MB/s |
| Active rebuild period total disk I/O | ~1.7 GB | ~1.7 GB |

Three things this measurement settled:

**1. The workload is genuinely disk-I/O-bound, not in-cache page-fault-bound.** The earlier "minor page faults dominate" hypothesis (based on 22 MB/s avg over the 100M trace) was wrong. The 22 MB/s was diluted by idle/wait time at the end. The active rebuild sustains 100-300 MB/s of real disk activity, and the iostat MB/s exactly matches the vm_stat pageins rate × page size — confirming the I/O is faulted-in pages, not idle minor faults.

**2. Caching does not help.** Warm rebuild back-to-back has identical disk I/O to cold. This means the I/O pattern is structural to the rebuild, not a one-time cold-load cost. More RAM would not change this.

**3. The SSD has large headroom but we're not using it.** Peak 327 MB/s is ~7% of Apple Silicon internal NVMe sequential bandwidth (~5 GB/s), 12 K tps is ~2% of typical NVMe random IOPS ceiling (~500 K). The disk is not the bottleneck — the *access pattern* is.

### What the I/O is (write amplification)

The 10M store on disk is ~1.1 GB total. Useful work for the rebuild is roughly:
- Read GSPO scan (~200 MB)
- Write GPOS index (~150 MB)
- Write trigram index (~250 MB)
- Total useful: ~600 MB

Actual measured: ~1.7 GB during the active period. **Roughly 3× the useful amount.**

That ~3× factor is the signature of B+Tree random-insert write amplification: each leaf page is read, modified, and written back every time a key destined for that page arrives. With 10M atoms scattering across the secondary index, the same leaf gets touched many times — each touch is a separate read + write. Trigram has the equivalent problem: each posting list page gets read + appended + written every time another atom hashes to its bucket.

This is exactly the inefficiency Phase 5.1.c sort-insert was meant to eliminate for GPOS. It got shipped but didn't deliver wall-clock because the sort implementation (`Array.Sort` with comparator on a 32-byte composite key, ~2.67 B comparisons at 100M, with a 3.2 GB buffer + `List<T>` growth + `ToArray` + drain — all serialized) cost as much as the random-walk savings. The trigram path was never sort-fixed at all.

## Decision

Phase 5.2 produced a layered finding:

- **Phase 5.1.b/c "neutral" call was wall-clock-only and missed structural overhead.** 1.7.37 generates allocation pressure (453 s finalizer time) and lock chatter (552 s `Monitor.Enter_Slowpath`) that 1.7.34 has zero of. Hardware absorbed it at 100M but the cost is real and scales linearly.
- **The fundamental bottleneck is write amplification, not CPU and not raw disk bandwidth.** ~3× more disk I/O than the rebuild's useful work requires. The SSD is at ~7% of bandwidth and ~2% of IOPS — the access pattern is the problem.
- **Sort-insert was the right idea, the wrong implementation.** The concept eliminates write amplification by touching each leaf once. The execution cost more than it saved.

**Recommended order of operations:**

1. **Revert parallel rebuild + sort-insert to 1.7.34 baseline.** Removes the GC/lock overhead and frees the design space for the right sort implementation.
2. **Implement radix external sort for GPOS rebuild.** 32-byte composite keys are radix-sortable byte-by-byte without a comparator (5-10× faster than `Array.Sort` with comparator at 100M scale), and an external streaming sort avoids the 3.2 GB monolithic buffer.
3. **Apply the same pattern to the trigram path.** Today trigram is purely random-write-amplified; today's largest source of structural I/O cost. Sort trigram entries by `(triagram, atom)` before inserting so each posting list is appended to once.
4. **Validate at 100M and 1B before committing.** The 100M measurements should show the I/O drop from ~3× to closer to 1× useful, and wall-clock should drop proportionally.

**Hardware constraint to note:** RAID is not available on this hardware (Apple Silicon, soldered internal NVMe, no add-in cards). Even if it were, it would help only proportionally with the *current* I/O volume — which is mostly amplification, not useful work. Eliminating amplification is the larger lever and must come first regardless.

For the 24 h Phase 6 exit at 21.3 B (linear extrapolation from current 100M = ~30 hours, super-linear past RAM probably worse), the path is: revert + radix external sort applied to both indexes. The Phase 3 follow-up list (parallel sort, etc.) is deferred — neither parallelism nor more sophisticated comparator-sort variants address the actual bottleneck.
