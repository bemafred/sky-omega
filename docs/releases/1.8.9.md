# Sky Omega 1.8.9 — DrHook debug-state visualization: a window into the agent's debugging

**Released:** 2026-06-26
**Version bump:** `Directory.Build.props` → 1.8.9
**Predecessor:** [Sky Omega 1.8.8](1.8.8.md) — DrHook: the runtime-version lock (and living with a preview SDK).

## Headline

DrHook's distinguishing capability over a classical debugger is that the agent debugs with a **`(hypothesis, observation)` braid** — it states what it expects before it looks, then records what it saw. 1.8.9 begins making that loop **legible to a human, live**: the engine now publishes a unified, surface-agnostic **debug-state model** over a Unix-domain **rendezvous socket**, and a standalone view renders it as the agent steps. The first view is a console (`drhook-viz-console`); a TUI and an Avalonia GUI are later phases over the *same* client and model.

This is [ADR-012](../adrs/drhook/ADR-012-debug-state-surfaces.md) (Phases 1–2). The MCP debugger stays **owned by the LLM**; views are **standalone, human-launched** peers that connect mid-session and are read-only at this phase. Terminating a view never affects the ongoing session; ending the session cleanly drops the views. It was **dogfooded end-to-end through the real DrHook MCP** — which is the point: visualization is a deliberately low-stakes vehicle for exercising DrHook on its own substrate.

## The architecture — one client, many views

The shape is the load-bearing decision, and it mirrors Mercury's `Mercury.Abstractions ← Mercury.Sparql.Tool ← Mercury.Cli.Sparql` exactly, so the console, the eventual TUI, and the eventual GUI share one client and one model — and no view ever drags in the engine's native (ICorDebug / dbgshim / Roslyn) dependencies:

| Layer | Project | Scope |
|-------|---------|-------|
| **Protocol contract** | `DrHook.Wire` (89 lines) | The NDJSON wire DTOs + a source-generated `WireCodec` (`Serialize` / `Parse`) + `WireRendezvous` (the shared socket path). Zero-dependency; **both** the server and every view depend on it, so the wire format is a deliberate contract decoupled from internal domain records. |
| **Client library** | `DrHook.Viz` (176 lines) | `DebugStateClient` (connect → read NDJSON → parse → update model → push to a view), `DebugStateClientModel` (current snapshot + a bounded delta tail), `IDebugStateView` (the seam views implement — symmetric with the engine's `IDebugEventSink`). BCL-only; references **only** `DrHook.Wire`. |
| **View** | `DrHook.Viz.Console` (100 lines) | `drhook-viz-console` — a thin `ConsoleDebugStateView` over a `TextWriter` that tails the live debug-state (a compact block per snapshot, one line per delta). The simplest view; the first proper DrHook debuggee. |

### The server side

`DrHook.Engine/Transport/DebugStateServer` binds the rendezvous socket and publishes a **snapshot on connect** then the live **delta stream**; `DebugStateWireMapper` maps the domain model to the wire DTOs. `EngineSteppingSession` adds the server as a **4th `CompositeEventSink` member** and publishes a snapshot after each stop (`PublishTransportSnapshot`). The correctness property that shaped it: the **request-thread driver** (the thread that owns stepping) captures the immutable snapshot after each stop and pushes it — the transport **never calls into `DebugSession`**, so there is no transport↔stepping concurrency hazard. Everything is best-effort: a bind failure surfaces as an anomaly, a publish fault is swallowed, and debugging never breaks. The listener is per-session (started on launch/attach, stopped on session end); `DebugStateServer` was made **restartable** for that lifecycle.

### The rendezvous

A fixed, well-known Unix-domain-socket path — `~/Library/SkyOmega/drhook/session.sock` on macOS, `~/.local/share/sky-omega/drhook/session.sock` elsewhere (Windows AF_UNIX is a later refinement). One active session at a time, so a fixed path is the whole rendezvous: no broker, no registry. The socket is created on launch/attach and unlinked on session end.

## Dogfooded end-to-end

Validated live, through the real DrHook MCP, in two parts:

- **The view watches a live session.** `drhook_launch` a target → the engine bound the socket and published; a **separate `drhook-viz-console` process** connected and rendered the snapshot-on-connect (`pid`, runtime, `Worker.Compute @ Program.cs:27`, locals, args), then a live delta and a fresh snapshot when the agent continued (`n=1` → `n=2`), then a clean `disconnected` when the session ended.
- **The view is itself a debuggee.** `drhook_launch` the console view → breakpoints bind in its top-level async `Main`, locals are readable — the first proper DrHook debuggee.

The dogfood also earned its keep as DrHook hardening: stopping a session *at* a breakpoint surfaced the known **break-stopped-dispose** anomaly cluster (a `WorkerSilentBreak`, a terminated-process HRESULT, two late `mscordbi` callbacks) — all in the engine's dispose path, none in the transport, surfaced cleanly without a crash. An ADR-007 data point, captured because we *used* DrHook rather than avoided it.

## Resolved (ADR-012 open questions)

- **Q1 — topology:** MCP-server-as-host, no broker. The session-owning process is the listener; views are connect-in clients.
- **Q4 — wire format:** newline-delimited JSON via `System.Text.Json`, through a thin wire-DTO layer (so the contract is decoupled from internal record shapes / renames).
- **Q6 — dashboard lifecycle:** attach-only. Views are ephemeral; the session is LLM-owned.
- **Q7 — rendezvous:** a fixed well-known Unix-domain-socket path, one active session at a time.
- **Q3 (lifecycle half):** a view's lifecycle is independent of the session's; terminating a view never affects the ongoing session.

## What's next

Phases 3–6 remain, each a new view or capability over the **same** `DrHook.Viz` client + `DrHook.Wire` contract: recording the `(hypothesis, observation)` braid in the model (Phase 3), a full read-only TUI (Phase 4), bidirectional control — commands in (Phase 5), and the Avalonia GUI sibling (Phase 6). Q2 (TUI technology), Q5 (PTY-in-surface), and the command half of Q3 are still open.
