# Sky Omega 1.8.10 — DrHook source-on-step views + typed value rendering

**Released:** 2026-06-27
**Version bump:** `Directory.Build.props` → 1.8.10
**Predecessor:** [Sky Omega 1.8.9](1.8.9.md) — DrHook debug-state visualization: a window into the agent's debugging.

## Headline

1.8.9 began publishing a surface-agnostic **debug-state model** to standalone views. 1.8.10 makes those views show **what you'd actually want when stepping**: the *source code* you're stopped in, and the *types* of the values — not just `Type.Method @ file:line` with a bare `?` for every object. This is [ADR-012](../adrs/drhook/ADR-012-debug-state-surfaces.md) **Phase 4** (now **Accepted** — the approach is validated; the full TUI surface + Q2 remain the engineering).

It was **dogfooded end-to-end three times through the real DrHook MCP** — which is the point: visualization is a deliberately low-stakes vehicle for exercising DrHook on its own substrate, and this cut earned two real findings that were fixed to completion in the same session.

## Source-on-step

The execution position used to reach a view only as the flattened `"Type.Method @ basename:line"` strings of the call stack — a view could show *where* execution stopped but couldn't *open* the file (the engine resolved the full path, then `Path.GetFileName` discarded it). A **Phase-2 enrichment** fixed that lossy wire: a new `FrameLocation` / `WireFrame` carries the **full** source path + line per frame (`WirePosition.CallStack` is now structured, mirroring how breakpoints already carried structured `File`/`Line`).

On that foundation, a **view-agnostic source-window primitive** in `DrHook.Viz`:

| Piece | Role |
|-------|------|
| `SourceWindowReader` | Resolves a `WireFrame`'s `File`+`Line` to a windowed render-model by reading the file from disk — sound because the transport is a **local** Unix-domain socket, so a view is co-located with the source tree. Bounded-LRU `SourceFileCache` (file-count + byte budgets); best-effort **named statuses** (`FileNotFound` / `FileTooLarge` / `LineOutOfRange` / `NoLocation`), never an exception. |
| `SourceWindow` / `SourceLine` | The render-model — lines around the current line, which one is current. Not styled text, so it is **independent of Q2** (the same primitive feeds a BCL-Console TUI, a library TUI, or the Avalonia GUI). |

`DrHook.Viz.Console` now renders a **source pane** between the frame and locals lines:

```
  frame   : Worker.Compute @ Program.cs:29
  source  : Program.cs
      27          long contribution = doubled + label.Length;
      28          var tags = new List<string> { label };
  ►   29          Total += contribution;   // SNAPSHOT_HERE
      30          GC.KeepAlive(tags);
  locals  : doubled=2, contribution=6, tags={List<String>}
  args    : this={Worker}, n=1, label=tick
```

## Typed value rendering — two dogfood findings, fixed to completion

Dogfooding surfaced two things; neither was deferred (a substrate requires feature completeness at introduction):

- **`this=?` → `this={Worker}`.** Object references were lossy on the wire — `WireVar` dropped the "is this an expandable object?" signal, so the view couldn't tell a non-null object from a null value and fell back to `?`. `WireVar` gained `HasChildren`, made **null-aware** (a null reference now renders `null`, not a misleading `{…}`).
- **Runtime type names, including generics.** A new `ValueTypeInspector` resolves a value's runtime type (`ICorDebugValue2.GetExactType` → the shared `NameOfType` → `MetadataResolver.TypeNameFromToken`), detects null references (`ICorDebugReferenceValue.IsNull`), and walks `EnumerateTypeParameters` **recursively** for generics — so `this={Worker}`, `tags={List<String>}`, `Dictionary<String, Foo>`, `Bar[]`. The full namespace-qualified name travels on the wire (the substrate is complete); the view abbreviates each component for display. `TypeName` threads through **every** value carrier — args, locals, fields, array elements → wire → mapper → the MCP `locals` JSON → the view.

A convergence note: `ExceptionInspector` had its own copy of the `ICorDebugType → type name` step; it was **deleted and pointed at the one shared `NameOfType`** rather than letting a second implementation drift.

## Dogfooded end-to-end

Validated live, through the real DrHook MCP, three times — each after reconnecting the MCP so the engine picked up the change being tested:

- **Source pane on a real target, tracking stepping.** `drhook_launch` → break → a **separate `drhook-viz-console` process** rendered the real `Program.cs` around the stopped line (`►` on it), and re-rendered as the agent continued (`n=1` → `n=2`). This settled the question the unit tests couldn't: the PDB's `File` path resolves to **readable on-disk source**.
- **`this={Worker}`** — the receiver's runtime type, resolved from the target's metadata, rendered live.
- **`tags={List<String>}`** — a generic local (the integration target gained a `List<string>` so the path is exercised every CI run).

Clean teardown each time: `drhook_stop` (graceful), the view reported a clean `disconnected`, the socket unlinked, **no orphans**.
