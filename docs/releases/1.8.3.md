# Sky Omega 1.8.3 — DrHook lifecycle, console isolation, and the 22-tool surface

**Released:** 2026-06-06
**Version bump:** `b1c5c18` (`Directory.Build.props` → 1.8.3)
**Predecessor:** [Sky Omega 1.8.2](1.8.2.md) — netcoredbg retired; DrHook substrate-independence reached

## Headline

With the substrate independent at 1.8.2, 1.8.3 is the first release to *shape the DrHook surface for its actual user* — an AI reasoning loop, not a human at an IDE. Three things land: the MCP tool surface is renamed to established IDE-debugger convention (ADR-010 Tier 1); the session lifecycle gains an explicit `stop` / `detach` / `kill` triad with a launched debuggee's console output isolated from the MCP channel (ADR-011); and the lifecycle is now complete for both Owned (launched) and Borrowed (attached) targets (findings F-010-1, F-010-2). One Mercury correctness fix rides along: aggregation dropped on the top-level GRAPH execution path.

## DrHook — tool-surface redesign (ADR-010 Tier 1, Accepted)

- MCP tools renamed to VS / VS Code / Rider vocabulary, matching the substrate's own `DebugSession.Launch` / `.Attach` verbs: `drhook_launch` (was `drhook_step_run` — launches a NEW process), `drhook_attach` (was `drhook_step_launch` — attaches to a running PID), `drhook_stop` / `drhook_continue` / `drhook_pause`, `drhook_step_over` / `_into` / `_out`, the `drhook_break_*` family (source / function / exception / list / remove / clear), and `drhook_locals`. The inverted internal adapter (the launch-named method attached, and vice-versa) is corrected. For an agent that selects a tool by *reading its name*, an inverted name is a correctness defect, not a cosmetic one.
- `drhook_step_test` removed — it only ever returned a structured "not implemented." Project-aware launch from a `.csproj` is the planned replacement (ADR-010 Tier 3); until then, attach to the testhost child with `drhook_attach`.
- Stale `netcoredbg` / DAP strings removed from tool descriptions.

## DrHook — lifecycle and console I/O (ADR-011, Accepted)

- **Lifecycle triad** `drhook_stop` / `drhook_detach` / `drhook_kill` (D1). `kill` is filed as an anomaly path, not a cleanup step — a well-built process ends on its own, so reaching for force is always worth recording. All three require a `hypothesis` (Decision principle 5); for `kill`, the reason force was needed is echoed back.
- **Lifecycle complete for both ownership models.** `drhook_kill` force-terminates Borrowed (attached) targets (F-010-1); `drhook_detach` leaves an Owned (launched) target running via `DetachLeaveRunning` with disown-and-reap (F-010-2). Both were "pending finding" at the start of the cycle; both close here. Detach refuses outstanding breakpoints (deactivate first).
- **Console isolation (D2, macOS).** `drhook_launch` owns process creation — `posix_spawn` the target SUSPENDED with stdout/stderr redirected to DrHook-owned pipes, then `RegisterForRuntimeStartup` + `SIGCONT` — so a debuggee's `Console.WriteLine` can no longer corrupt the MCP JSON-RPC channel (probe 59 / finding 75). A launch-time environment override closes the silent MCP→substrate drop (probe 63 / finding 81).
- **Output surfacing (D3).** `drhook_drain_console` + `drhook_drain_log` surface a launched debuggee's captured stdout/stderr and logpoint output via a surface-agnostic `IDebugEventSink.OnConsoleOutput` + `CompositeEventSink` — the seam ADR-012's future views consume. Console / log / anomaly buffers reset at session start, so drains reflect only the current session.

This brings the live MCP surface to **22 tools**.

## DrHook — breakpoint and inspection fixes

- Full pending-breakpoints, so breakpoints bind in lazily-loaded modules; an entry-module hold-gate so `drhook_launch` works without a `Debugger.Break`.
- Source breakpoints bind when the source file name differs from the assembly name (probe 60 / finding 76).
- `drhook_continue` no longer NREs when the debuggee exits during the continue (probe 61 / finding 77).
- `drhook_locals` serialization fixed — typed `JsonValue` dispatch for boxed `RawValue`.
- Detaching an already-terminated target no longer emits a spurious anomaly.

## Mercury — one correctness fix

- **Aggregation no longer dropped on the top-level GRAPH execution path** (`b1c5c18`). Solution modifiers are threaded through `FromMaterializedWithGraphContext` and the `QueryExecutor` call site, and the shared materialized-with-graph-context constructor now sets `_isMaterialized` so the grouped-collection path scans materialized rows instead of an uninitialized scan. Because that constructor is shared with `FromMaterialized`, the same latent subquery-aggregate case is closed. **Conformance-blind** — the W3C suite exercises default-graph aggregation only — and surfaced within minutes while dogfooding semantic-memory recall, where all data lives in named graphs. +4 `QueryExecutorTests.GraphAggregation` tests (RED before, GREEN after); 421/421 SPARQL 1.1 Query conformance held.

## Substrate posture at 1.8.3

- `DrHook.Engine`: **6,231** lines hand-written `.cs` (BCL + P/Invoke + source-gen COM, single managed dependency, native asset via per-RID NuGet) — up from 4,071 at 1.8.2.
- `DrHook.Mcp`: **1,566** lines; **22 MCP tools** — up from 972 lines / "17-method" at 1.8.2.
- `DrHook.Engine.Tests`: **89** test methods — up from 47.
- PoC probes numbered through **63**; **81** finding documents; **172** fixture files in `poc/drhook-engine/`.
- Mercury substrate: the GRAPH-aggregation fix; otherwise unchanged from the 1.8.0 cutover. Full suite **4,526** (4,522 + 4 GRAPH-aggregation regression tests).

## What 1.8.3 does NOT carry

- **ADR-012 debug-state surfaces** (Proposed): the surface-agnostic debug-state model and its first human views (TUI dashboard, Avalonia sibling). The D3 sink seam is in place; the views are not built.
- **Cross-platform validation:** still macOS/arm64 only. `DrHook.Mcp` bundles all 7 RIDs; per-platform probe runs are ADR-007 Phase 9.
- **ADR-007 production-suitability:** teardown + concurrency hardening, substrate-aligned test-runner debugging, integration-test mechanism characterization — all open (ADR-007, Accepted; sequencing not yet executed).

## What's next

[ADR-012](../adrs/drhook/ADR-012-debug-state-surfaces.md) (Proposed) takes the surface-agnostic debug-state seam built in D3 and drives a terminal dashboard and an Avalonia sibling from it — one live picture of a stopped program, multiple views: an agent-first IDE, with the human looking over the agent's shoulder rather than the other way around. [ADR-007](../adrs/drhook/ADR-007-teardown-concurrency-test-debug.md) (Accepted) sequences the production-suitability work that makes DrHook safe beyond a single developer's macOS box.
