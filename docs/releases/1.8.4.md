# Sky Omega 1.8.4 — Mercury convergence: one execution path, one expression evaluator

**Released:** 2026-06-21
**Version bump:** `ca66c1e` (`Directory.Build.props` → 1.8.4)
**Predecessor:** [Sky Omega 1.8.3](1.8.3.md) — DrHook lifecycle, console isolation, the 22-tool surface

## Headline

1.8.4 is the Mercury **convergence** release — the first since 1.8.3, and where the divergence-register work lands in a shipped build. Everything in it collapses two implementations into one: the SPARQL *execution* path (ADR-047), the cross-format RDF *canonicalizers* (divergence S1), and the SPARQL *expression* evaluator (ADR-049 / divergence S2). The marquee is the last of these — FILTER and BIND had drifted into two independent recursive-descent evaluators, each implementing a non-conformant *half* of the SPARQL `[110] Expression` grammar. They are now one evaluator; the duplicate is deleted.

## Mercury — one SPARQL expression evaluator (ADR-049, divergence S2 closed)

SPARQL 1.1 defines a single `[110] Expression` production that both FILTER and BIND consume; the only spec-level difference is what happens *after* evaluation — FILTER takes the effective boolean value (§17.2.2), BIND binds the produced term (§18). Mercury had drifted from that: `FilterEvaluator` had the full logical grammar and 52 functions but **no arithmetic**; `BindExpressionEvaluator` had arithmetic but **no `||`/`&&`/`!`** and lacked the boolean predicates. Each was a different non-conformant half.

1.8.4 makes `FilterEvaluator.EvaluateToValue` the **one** `[110]` evaluator. FILTER derives its boolean via `CoerceToBool(EvaluateToValue(...))`; BIND binds the produced term, leaving the variable unbound on error. `BindExpressionEvaluator` is **deleted (2,336 lines)**.

The merge was strictly incremental — the full W3C suite stayed green after every step, **zero reverts**:

1. **Function arguments through the full grammar.** The function library parsed each argument with a weak single-term parser, so a compound argument (`COALESCE(1 + 1, 0)`) left the parser parked on the operator and the variadic argument loops spun forever — a latent FILTER hang too. Arguments now parse through the full value grammar.
2. **Output-form punch-list.** `NOW`/`STRUUID`/`TZ` quoting, `NOW`/`TIMEZONE` typed literals, `UUID`/`IRI`/`URI` angle-bracketing (which also fixed a latent FILTER bug — `IRI(x)` now compares equal to the literal `<x>`), `CEIL`/`FLOOR`/`ROUND` datatype preservation, `SECONDS` full lexical precision, `BNODE` per-row seeding.
3. **Literal parsing.** `ParseStringLiteral` now preserves `@lang` and non-numeric datatypes (it had dropped them).
4. **Base-IRI parity**, then the actual wiring: the 9 BIND / SELECT-expression / GROUP-BY-expression call sites routed to `EvaluateToValue`, FILTER switched to `CoerceToBool(ValueOr())`, and the ~165-line dead bool grammar removed.

### The reconciliation lesson — both evaluators were wrong, in different places

Routing BIND through `FilterEvaluator` exposed that the divergence was about **correctness, not just coverage** — and that *neither* implementation was a clean oracle:

- `FilterEvaluator`'s date/time accessors (`YEAR`…`SECONDS`, `TZ`, `TIMEZONE`) parsed via `DateTime.TryParse`, which **converts a timezone offset to machine-local time** — a wrong, non-deterministic component (`HOURS` of `…15:38:02-08:00`). Now they extract components directly from the lexical form. Its `STRDT`/`STRLANG` mishandled the datatype IRI and the lang-tag check; its `DATATYPE` collapsed decimal/double/integer because it read the value type instead of the preserved lexical datatype.
- The old `BindExpressionEvaluator` was the buggy side for `STRDT` with an inline IRI datatype (it double-bracketed, `^^<<…>>`) and for `LANG` (it returned unbound where the spec wants the tag / `""`).

The **spec** — not either implementation — was the standing oracle; both were corrected toward it. These divergences were invisible to a constant-inline probe and only surfaced under bound variables and non-Z offsets, which is why they had persisted. The two demonstrated conformance gaps are now locked as permanent tests (`Filter_WithArithmetic_Evaluates`, `Bind_WithLogicalOr_Evaluates`), and the differential reconciliation probe is frozen to explicit conformant literals now that its oracle is gone.

## Mercury — one SPARQL execution path (ADR-047)

Also shipping for the first time in 1.8.4 (landed on `main` after the 1.8.3 release): the default-path cutover that makes the unified `TreeJoinExecutor` the **sole** SPARQL execution path. All four default-path carve-outs — VALUES+triple, sub-query, FROM, SERVICE — were routed through the tree, then the **entire old slot-operator execution engine was deleted (~14,000 lines gross)**: `QueryExecutor.{Service,Subquery,Exists}.cs`, the streaming `QueryResults` machinery, and the operator cluster (`MultiPatternScan`, `CrossGraphMultiPatternScan`, `DefaultGraphUnionScan`, `VariableGraphScan`, `SubQuery*`, `BoxedSubQueryExecutor`, `SlotBasedOperators`, `IScan`, `ScanType`). The last property-path conformance skip, `seq-alt-of-star`, is closed. `ServiceMaterializer` is spared for a future federation-efficiency effort ([ADR-048](../adrs/mercury/ADR-048-service-federation-efficiency.md), Proposed). See [ADR-047](../adrs/mercury/ADR-047-default-path-cutover.md) (Completed).

## Mercury — cross-format canonicalizer convergence (divergence S1)

The format parsers had each grown their own escape decoding, relative-IRI resolution, and numeric/boolean xsd typing. S1 converged all of them onto shared `RdfEscape` / `RdfIri` / `RdfNumeric` in `Mercury/Rdf` (commits `afd73f8`→`8152f17`), retiring the duplicated logic across JSON-LD, Turtle, TriG, RDF/XML, N-Triples, and N-Quads — `Rdf` grew 536 → 901 as the shared canonicalizers moved in, and the format folders shrank net ~800 lines.

## Substrate posture at 1.8.4

- Mercury source **82,539 → 79,179** lines. `Sparql` **39,160 → 36,269** (ADR-049). `Rdf` **536 → 901** (S1 shared canonicalizers).
- Full Mercury test suite **4,699 / 0 failed / 6 skipped** (the 6 are JSON-LD); **4,705** test cases.
- W3C conformance **unchanged** — the unification preserved every result: SPARQL 1.1 Query 421/421, Update 94/94, core formats 1,181/1,181, JSON-LD 461/467.
- Substrate invariants intact: **26 public Mercury types, BCL-only core**.
- DrHook substrate unchanged from 1.8.3 (the lifecycle/console/22-tool surface); this is a Mercury release.

## What 1.8.4 does NOT carry

- **No cognitive-layer work.** Lucy / James / Mira / Sky remain ahead on the 1.8.x roadmap; 1.8.4 is substrate convergence, not a cognitive-layer release.
- **No `ServiceMaterializer` change.** SERVICE federation efficiency is [ADR-048](../adrs/mercury/ADR-048-service-federation-efficiency.md) (Proposed) — the spared materializer is its starting point.
- **No new W3C coverage.** The unification was conformance-preserving by construction; it adds correctness on shapes the corpus does not exercise (FILTER arithmetic, BIND `||`, date components under offsets), locked by Mercury tests rather than the W3C suite.

## What's next

The divergence register's two large SPARQL-side splits (S1 canonicalizers, S2 evaluators) and the execution-path split (ADR-047) are now closed. Remaining register items are smaller and format-local. The 1.8.x line continues toward the cognitive-layers entry point per the [version-line model](../roadmap/production-hardening-1.8.md).
