# Sky Omega 1.8.8 — DrHook: the runtime-version lock (and living with a preview SDK)

**Released:** 2026-06-25
**Version bump:** `Directory.Build.props` → 1.8.8
**Predecessor:** [Sky Omega 1.8.5](1.8.5.md) — DrHook navigable inspection + the Mercury silent-drop it surfaced. (1.8.6 and 1.8.7 were version bumps without standalone notes: **1.8.6** — DrHook ADR-014 size-checked value reads; **1.8.7** — Mercury undefined-prefix error + DURING graph-scoping fix + the Lucy recall skill.)

## Headline

A single-file app "wouldn't debug" through the live DrHook MCP — it failed with the opaque `HRESULT 0x80131C3C`. The obvious read was "single-file is unsupported." It was wrong. The real cause is a property of how .NET debugging works: **a debugger process can host the debug components (`mscordbi`/DAC) of only ONE runtime major at a time** — the first target it debugs loads them process-globally, and they never unload. The long-lived MCP had debugged a **net11** file-based app first this session, so every **net10** target afterward — the single-file app among them — failed. Not single-file. Not "same version is unsupported" either (a tempting correlation we explicitly disproved). 1.8.8 root-causes it, ships a guard that turns the cryptic failure into a one-line instruction, and pins the SDK so the trigger stops happening.

## The investigation — mechanism, not correlation

`0x80131C3C` decodes to **`CORDBG_E_DEBUG_COMPONENT_MISSING`** — dbgshim couldn't load a debug component. Yet every component was present on disk (10.0.x and 11.0 both ship `libmscordbi`/`libmscordaccore`). The discriminator was the *target's* runtime version: net11 targets worked through the MCP all session; net10 targets failed. A seductive hypothesis — "a debugger can't debug a target on its own runtime version" — was **disproved** (a net11 host debugged a net11 target fine). We nailed the real mechanism with two pieces of evidence:

- **`lsof` on the live server**: it ran on its own 10.0.9 runtime but had the **11.0** `libmscordbi` + `libmscordaccore` + `libcoreclr` resident (pulled in debugging net11 apps earlier) and **no 10.0 `mscordbi`** — and that 11.0 lock **persisted** across session teardown and every failed net10 attempt.
- **A one-process, two-version reproduction**: `Launch(net11)` → OK; then `Launch(net10)` in the same process → `0x80131C3C`. One version per process always works; mixing them never does.

Why it bit us: the substrate targets **net10.0**, but a file-based app (`dotnet run x.cs`) under the installed **.NET 11 preview SDK** builds **net11.0**. Debug a file-based probe, then a net10 target, and the process is locked the wrong way.

## The fix — detect + guide

DrHook now refuses a mismatched launch/attach **cleanly and up front**, instead of failing late inside dbgshim and leaking a suspended spawn:

- **`RuntimeConfig`** reads a target's `runtimeconfig.json` (`tfm` / `framework.version`) to learn its runtime major **before** dbgshim spawns or attaches — a pure file read, unit-tested.
- **`DebugSession`** records the first-debugged major process-globally, and **`GuardRuntimeLock`** refuses a mismatch with an actionable message — *"This DrHook debugger process is locked to .NET 10 … cannot also debug a .NET 11 target … reconnect, or pin this target to net10.0"* — **no opaque HRESULT, no leaked process**.
- The locked runtime is **surfaced**: a `lockedRuntime` field on the launch/attach response and `DebugSession.ProcessRuntimeMajor`, so the operator sees what the server is bound to before hitting trouble.

Validated **live through the MCP** on a fresh server: a net10 target — the exact thing that failed all session — launches, stops, and inspects (`x=7`, `d=14`, `lockedRuntime ".NET 10"`); a subsequent net11 target gets the clear refusal, no `0x80131C3C`, no leak. 5 new `RuntimeConfigTests`; full `DrHook.Engine.Tests` **135 / 135**.

The deeper constraint — one `mscordbi` per process — is an ICorDebug platform reality; truly debugging mixed versions from one server would need a helper process per runtime major. The guard demotes that from *needed* to *nice to have*: a cryptic crash becomes a clear "reconnect."

## Living with the .NET 11 preview

The preview is installed deliberately (to test Sky Omega under .NET 11), but its presence silently changed build defaults — file-based apps build net11, `dotnet publish x.cs` defaults to NativeAOT. A repo-root **`global.json`** now pins the SDK to the latest **10.x** (`rollForward: latestMinor`, never 11), so every build and file-based app defaults to net10 again and DrHook only ever sees net10 targets. The preview stays installed for deliberate testing — override or remove the `global.json` for that. Verified: `dotnet --version` → 10.0.301; Mercury and DrHook build clean under SDK 10.

## Also in 1.8.8 — argument-name fidelity + managed single-file debugging

The arc that *preceded* the lock investigation (findings 82–85 — and the reason a single-file app was in the debugger at all):

- **Arguments are first-class everywhere.** Frame arguments carry their real source names (a new `MethodMetadata` reads the PE `Param` table + the `mdStatic` bit; `ArgumentValue` gained a `Name`), so `drhook_locals` / `drhook_expand` show `seed` / `this` / `d`, not `arg0` / `argN`; conditions and logpoint templates reference a parameter by name; and member-access on an argument receiver (`w.Area == 200`) func-evaluates the getter.
- **Managed `PublishSingleFile` apps debug.** A bundled assembly has no on-disk PE, so DrHook recovers symbols from the sidecar PDB (`DebugType=portable`) or by reading the loaded module's image from target memory and extracting the embedded PDB (`DebugType=embedded`); single-file argument names come from the loaded module's metadata via `IMetaDataImport`. NativeAOT remains structurally unsupported (no managed runtime, no ICorDebug) — and it is the silent default for `dotnet publish app.cs`.

DRHOOK.md gained a **"Lifecycle discipline & common pitfalls"** section this cycle — the runtime-version rule plus the broader operator guidance (no `Debugger.Break()` crutch; `drhook_stop`, not `drhook_kill`; clean up orphaned targets) — written because DrHook is the first debugger built for an LLM operator and there is no training corpus for driving one. The docs are the training surface.
